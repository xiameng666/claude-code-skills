# GMS 逆向分析完整工作流

## 概述

本文档描述了一套完整的 GMS 逆向分析工作流，结合了：
- **JEB MCP** - 逆向工程数据源
- **知识库系统** - SQLite + MD 文件持久化
- **Ralph Wiggum 机制** - 自引用循环确保行为持久化
- **多 Agent 协作** - 图论驱动的类依赖分析
- **编译验证** - 每步还原后确保代码可编译


## Phase 3: 图论分析 (Ralph 循环)

### 3.1 核心思想

**使用 Ralph Wiggum 机制确保分析行为持久化**：

```bash
/ralph-loop "
  以 Lfvxn; 为根节点，构建完整的类依赖图。

  目标: 遍历所有依赖直到到达叶子节点（不依赖其他混淆类的节点）。

  规则:
  1. 每个类必须分析方法体依赖（new、静态调用、类型转换）
  2. 系统类 (java.*, android.*) 不入图
  3. 每处理一个类，立即写入 MD 文件
  4. 使用双 Agent 模式: AgentA 分析 → AgentB 审查

  完成标准: 连续 2 轮遍历未发现新的混淆类
  完成时输出: <promise>GRAPH_COMPLETE</promise>
" --completion-promise "GRAPH_COMPLETE" --max-iterations 500
```

### 3.2 Ralph 状态文件

创建 `.ralph/graph-analysis.md` 持久化分析状态：

```yaml
---
active: true
iteration: 1
max_iterations: 500
completion_promise: "GRAPH_COMPLETE"
seed_class: "Lfvxn;"
started_at: "2026-03-01T10:00:00Z"
---

# 图论分析任务

## 种子类
Lfvxn; (推测: FusionEngine)

## 目标
构建以种子类为根的完整依赖图

## 分析队列
- [x] Lfvxn;
- [ ] Labc; (待分析)
- [ ] Ldef; (待分析)

## 叶子节点
(分析过程中填充)

## 完成条件
连续 2 轮遍历未发现新的混淆类
```

### 3.3 多 Agent 协作模式

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         双 Agent 协作流程                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │  Agent A: 分析者                                                     │   │
│   │                                                                      │   │
│   │  1. 从队列获取待分析类                                               │   │
│   │  2. 调用 JEB MCP 获取信息:                                           │   │
│   │     - get_class_decompiled_code (方法体 - 最重要!)                   │   │
│   │     - get_class_superclass (父类)                                    │   │
│   │     - get_class_interfaces (接口)                                    │   │
│   │     - get_class_fields (字段类型)                                    │   │
│   │     - get_class_methods (方法签名)                                   │   │
│   │  3. 提取依赖:                                                        │   │
│   │     - 直接依赖 (不需要审查)                                          │   │
│   │     - 方法体依赖 (需要重点审查!)                                     │   │
│   │  4. 生成分析结果，提交 Agent B 审阅                                  │   │
│   │                                                                      │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                    │                                        │
│                                    ▼                                        │
│   ┌─────────────────────────────────────────────────────────────────────┐   │
│   │  Agent B: 审查者                                                     │   │
│   │                                                                      │   │
│   │  审查检查项 (只审查方法体依赖):                                       │   │
│   │    □ new 语句是否全部提取?                                           │   │
│   │    □ 静态方法调用是否全部提取?                                       │   │
│   │    □ 类型转换是否全部提取?                                           │   │
│   │    □ Lambda/匿名类是否全部提取?                                      │   │
│   │    □ instanceof 检查是否全部提取?                                    │   │
│   │                                                                      │   │
│   │  审查结果:                                                           │   │
│   │    ✓ 通过 → 更新队列，继续下一个类                                   │   │
│   │    ✗ 不通过 → 说明遗漏项，Agent A 补充                               │   │
│   │                                                                      │   │
│   └─────────────────────────────────────────────────────────────────────┘   │
│                                    │                                        │
│                                    ▼                                        │
│                            从队列取下一个类                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.4 依赖分类

```
┌─────────────────────────────────────────────────────────────────┐
│  直接依赖（自动获取 - 不需要审查）                               │
├─────────────────────────────────────────────────────────────────┤
│  ✓ 父类 (get_class_superclass)                                  │
│  ✓ 接口 (get_class_interfaces)                                  │
│  ✓ 字段类型 (get_class_fields)                                  │
│  ✓ 方法签名类型 (get_class_methods)                             │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  方法体依赖（容易遗漏 - 必须审查！）                             │
├─────────────────────────────────────────────────────────────────┤
│  ⭐ new 语句中的类型                                             │
│  ⭐ 静态方法调用 (Xxx.a(), Xxx.b())                              │
│  ⭐ 方法链调用中的类型                                           │
│  ⭐ 类型转换 (Lxxx;) obj                                        │
│  ⭐ Lambda/匿名类                                                │
│  ⭐ instanceof 检查                                              │
│  ⭐ 异常处理 (catch, throws)                                    │
│  ⭐ 泛型类型参数                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.5 图论完成标准

```
┌─────────────────────────────────────────────────────────────────┐
│  完成判定                                                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  条件: 连续 2 轮遍历未发现新的混淆类                             │
│                                                                 │
│  伪代码:                                                        │
│    prev_queue_size = 0                                          │
│    while True:                                                  │
│      analyze_all_pending()                                      │
│      if len(pending) == 0:                                      │
│        break  # 队列为空，完成                                   │
│      if len(pending) == prev_queue_size:                        │
│        break  # 连续两轮大小不变，无新发现                       │
│      prev_queue_size = len(pending)                             │
│                                                                 │
│  输出: <promise>GRAPH_COMPLETE</promise>                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Phase 4: 逆向还原

### 4.1 核心策略：从叶子到根

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         逆向还原策略                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   依赖图结构:                                                               │
│                                                                             │
│                      Lfvxn; (根节点)                                        │
│                         │                                                   │
│              ┌──────────┼──────────┐                                       │
│              │          │          │                                        │
│            Labc;      Ldef;      Lghi;                                     │
│              │          │                                                   │
│           Ljkl;      Lmno;                                                 │
│              │                                                              │
│           Lpqr; (叶子)                                                      │
│                                                                             │
│   还原顺序 (拓扑逆序):                                                      │
│     1. Lpqr; (叶子 - 无依赖)                                                │
│     2. Ljkl; (依赖 Lpqr;)                                                  │
│     3. Lmno; (叶子)                                                         │
│     4. Labc; (依赖 Ljkl;)                                                  │
│     5. Ldef; (依赖 Lmno;)                                                  │
│     6. Lghi; (叶子)                                                         │
│     7. Lfvxn; (根 - 依赖所有)                                              │
│                                                                             │
│   优势: 每一步还原时，所有依赖类已存在，可以立即编译验证                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 4.2 Ralph 循环：逆向还原

```bash
/ralph-loop "
  逆向还原任务

  输入: 完成的依赖图 (Phase 3 产出)
  输出: 可编译的 Java 源码

  流程:
  1. 按拓扑逆序排列类
  2. 从叶子节点开始还原
  3. 每还原一个类:
     a. 生成 Java 文件到 generated/ 目录
     b. 运行 javac 编译验证
     c. 如果编译失败，修复错误
     d. 编译通过后，继续下一个类

  编译命令:
    cd {knowledge_dir}/generated
    javac -cp . com/google/android/gms/**/*.java

  完成标准: 所有类还原完成且编译通过
  完成时输出: <promise>REVERSE_COMPLETE</promise>
" --completion-promise "REVERSE_COMPLETE" --max-iterations 1000
```

### 4.3 编译验证流程

```
┌─────────────────────────────────────────────────────────────────┐
│  单类还原流程                                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 获取类信息                                                   │
│     └─→ get_class_decompiled_code()                             │
│     └─→ get_rename() (获取已重命名的符号)                       │
│                                                                 │
│  2. 生成 Java 源码                                               │
│     └─→ 应用重命名                                               │
│     └─→ 修复语法问题                                             │
│     └─→ 写入 generated/{package}/{Class}.java                   │
│                                                                 │
│  3. 编译验证                                                     │
│     └─→ javac -cp . {Class}.java                                │
│                                                                 │
│  4. 处理结果                                                     │
│     ├─→ 编译通过 → 标记完成，继续下一个                          │
│     └─→ 编译失败 → 分析错误，修复，重新编译                      │
│                                                                 │
│  5. 同步更新                                                     │
│     └─→ 更新 notes/{class}.md                                   │
│     └─→ 记录还原状态到 SQLite                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.4 编译错误处理

常见编译错误及修复策略：

| 错误类型 | 原因 | 修复策略 |
|---------|------|---------|
| 找不到符号 | 依赖类未还原 | 确保拓扑顺序正确 |
| 类型不匹配 | 泛型/类型推断问题 | 显式类型声明 |
| 方法签名冲突 | 重命名冲突 | 调整方法名 |
| 包不存在 | 包结构不完整 | 创建包目录 |
| 访问权限 | private/protected | 调整访问修饰符 |

### 4.5 增量编译策略

```bash
# 每次只编译新还原的类及其依赖
compile_incremental() {
  local class=$1
  local deps=$(get_dependencies "$class")

  # 先编译依赖
  for dep in $deps; do
    if [[ ! -f "${dep}.class" ]]; then
      compile_incremental "$dep"
    fi
  done

  # 再编译当前类
  javac -cp . "${class}.java"
}
```

### 为什么使用 Ralph 机制？

```
┌─────────────────────────────────────────────────────────────────┐
│  传统方式 vs Ralph 机制                                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  传统方式:                                                       │
│    - 依赖对话历史记忆                                            │
│    - 早期消息权重随距离衰减                                      │
│    - 可能"忘记"初始指令                                          │
│    - 长时间运行后行为漂移                                        │
│                                                                 │
│  Ralph 机制:                                                     │
│    - 状态持久化到文件                                            │
│    - 每次迭代重新读取完整指令                                    │
│    - 物理上不可能忘记                                            │
│    - 行为始终一致                                                │
│                                                                 │
│  本质: while true; do claude "$PROMPT"; done                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 总结

本工作流的核心创新：

1. **Ralph 机制** - 确保长时间运行时行为不漂移
2. **双 Agent 协作** - 提高依赖分析的准确性
3. **图论驱动** - 系统化遍历所有依赖
4. **拓扑逆序还原** - 保证每步都可编译验证
5. **增量编译** - 快速定位和修复错误

这套流程可以处理任意规模的混淆代码库，最终产出可编译的 Java 源码。
