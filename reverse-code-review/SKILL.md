## 核心原则

**所有对比基准必须通过 JEB MCP 获取真实反编译代码，禁止凭记忆推导或猜测。**

## 前置条件

- JEB 已加载目标 APK 且 MCP 连接正常
- 本地项目中的还原类包含 GMS 原名称注释（如 `原名称：xxxx` 或 `对应GMS: xxxx`）

## 工作流程

### Phase 1: 构建对比队列

#### 1.1 扫描本地类清单
- 递归扫描项目 `src/main/java` 下所有 `.kt` / `.java` 源文件
- **排除**测试类（`test`、`androidTest`）、UI 类（`Activity`、`ViewModel`、`Theme`）、纯 Demo 类
- **保留**所有 `com.google.*` 包下的还原类作为审计目标

#### 1.2 提取 GMS 映射
对每个还原类，从源码注释中提取：
- `原名称：xxx` → GMS 混淆类名
- `对应GMS: xxx` → GMS 语义类名
- 字段/方法注释中的 `GMS: xxx` → 成员级别映射

若无注释映射，标记为 `UNMAPPED`，跳过自动对比，记入报告待人工确认。

#### 1.3 按模块分组 & 排序
按依赖层级排序（叶子节点优先）：

```
优先级 1（基础工具类）: 数学工具、枚举、数据结构
优先级 2（模型/配置类）: Phenotype配置、数据模型
优先级 3（核心算法类）: 滤波器、粒子滤波、神经网络
优先级 4（业务编排类）: Fusion、Processor、Builder
优先级 5（顶层入口类）: Service、Engine、Dispatcher
```

### Phase 2: 逐类对比审计

对队列中每个类，执行以下对比步骤：

#### 2.1 结构对比
通过 `get_class_fields` 和 `get_class_methods` 获取 GMS 类信息：

| 检查项 | 对比方式 |
|--------|----------|
| 字段数量与类型 | 本地字段列表 vs GMS 字段列表，逐一匹配 |
| 方法签名 | 本地方法签名 vs GMS 方法签名，检查参数类型和返回值 |
| 继承关系 | 本地 extends/implements vs GMS 继承链 |
| 内部类 | 是否遗漏内部类或匿名类 |
| 访问修饰符 | public/private/protected 一致性 |

#### 2.2 算法对比（逐方法）
通过 `get_method_decompiled_code` 获取 GMS 方法实现，逐方法对比：

| 检查项 | 判定标准 |
|--------|----------|
| 控制流 | if/else/switch/loop 分支结构是否一致 |
| 数学运算 | 公式、常量、精度处理是否一致 |
| 数据结构操作 | 集合增删改查逻辑是否一致 |
| 边界处理 | null 检查、数组越界、溢出保护是否一致 |
| 异常处理 | try/catch 覆盖范围和处理逻辑是否一致 |

#### 2.3 回调与事件对比
| 检查项 | 判定标准 |
|--------|----------|
| 回调注册 | Listener/Callback 的注册时机和对象是否一致 |
| 回调触发条件 | 触发逻辑和参数传递是否一致 |
| 生命周期钩子 | init/destroy/reset 等时机是否一致 |
| 线程模型 | synchronized/volatile/锁机制是否一致 |

#### 2.4 运行逻辑对比
| 检查项 | 判定标准 |
|--------|----------|
| 调用链 | 方法 A→B→C 的调用顺序是否一致 |
| 状态机 | 状态转移条件和转移目标是否一致 |
| 数据流 | 输入→处理→输出的数据流转路径是否一致 |
| 副作用 | 全局变量修改、外部调用等副作用是否一致 |

### Phase 3: 产出审计报告

#### 3.1 报告结构

```markdown
# 逆向代码审计报告
> 审计时间: {timestamp}
> 审计范围: {module_list}
> 总类数: {total} | 通过: {pass} | 差异: {diff} | 未映射: {unmapped}

## 模块概览
{按模块汇总通过率的 mermaid 饼图}

## 逐类审计结果

### [类名] (GMS: {gms_name})
- 状态: ✅ 一致 / ⚠️ 有差异 / ❌ 严重偏离 / ⏭️ 未映射
- 结构对比: {结果}
- 算法对比: {逐方法结果}
- 回调对比: {结果}
- 逻辑对比: {结果}

#### 差异详情（如有）
| 方法 | 维度 | 本地实现 | GMS实现 | 严重程度 |
|------|------|----------|---------|----------|
| xxx  | 算法 | ...      | ...     | HIGH/MED/LOW |

## 待处理项汇总
- [ ] {需要修复的差异列表}
- [ ] {未映射待确认的类列表}
```

#### 3.2 严重程度定义

| 级别 | 含义 | 示例 |
|------|------|------|
| HIGH | 算法逻辑不一致，影响正确性 | 公式错误、分支遗漏、状态机缺失 |
| MED | 实现差异但不影响核心逻辑 | 边界值不同、异常处理缺失、精度差异 |
| LOW | 风格差异，不影响功能 | 变量命名、代码顺序、冗余检查 |
| INFO | 已知简化或占位 | 注释标注的 TODO、未实现的分支 |

## 文件输出规范

### 审计报告
- 输出路径: `{项目根目录}/docs/audit-report.md`
- 每次执行审计时覆盖写入，文件头包含时间戳用于区分版本

### 中间状态（断点续审）
- 对比队列 & 进度: `{项目根目录}/docs/audit-queue.md`
- 格式: 每行一个类，前缀标记状态
  - `[x] ClassName (GMS: xxx) → ✅` 已完成
  - `[ ] ClassName (GMS: xxx)` 待对比
  - `[-] ClassName → UNMAPPED` 无映射跳过
- 每完成一个类的对比后，立即更新此文件并将结果追加写入 `audit-report.md`，避免上下文丢失需要重跑

## 注意事项

- **断点续审**: 若执行中断，下次启动时读取 `audit-queue.md` 恢复进度，跳过已标记 `[x]` 的类
- 若 JEB MCP 返回 `Class not found`，尝试用混淆名和全限定名分别查询
- 对比时忽略：日志语句、纯注释差异、import 顺序、Kotlin/Java 语法糖差异
- 本地有意简化的部分（注释中标注了 TODO 或「简化处理」），标记为 INFO 而非差异
